## Goal

Add the high-level hosted evaluation dataset bridge for `PYD-3896`.

The end state is a TypeScript API on `LogfireAPIClient` that can:

- push a local `logfire/evals` `Dataset` instance to hosted Logfire datasets;
- fetch a hosted dataset export and materialize it as an executable local `logfire/evals` `Dataset`;
- keep low-level hosted dataset JSON operations separate from high-level evaluation dataset object construction.

This PRP builds on `PYD-3528`, which introduced the hosted datasets API client foundation in `logfire/datasets` and `@pydantic/logfire-node/datasets`.

## Why

- JS now has local evaluation primitives under `logfire/evals` and hosted dataset CRUD/import support under `logfire/datasets`, but users still have to manually translate between the two.
- Python already exposes high-level hosted dataset helpers through `LogfireAPIClient.push_dataset(...)` and typed `LogfireAPIClient.get_dataset(...)`.
- JS users need a TypeScript-readable equivalent that preserves Python behavior while avoiding confusing overloads between raw hosted JSON and executable local `Dataset` objects.
- This completes the main hosted companion workflow for local JS eval datasets without pulling in production trace/query dataset construction from `PYD-2417`.

## Success Criteria

- [ ] `LogfireAPIClient` exposes `pushEvaluationDataset(dataset, options)` for local `logfire/evals` `Dataset` instances.
- [ ] `LogfireAPIClient` exposes `getEvaluationDataset(idOrName, options)` for fetching hosted data into a local `logfire/evals` `Dataset`.
- [ ] Existing low-level methods such as `getDataset(...)`, `createDataset(...)`, and `addCases(...)` continue to return/accept raw hosted dataset API JSON.
- [ ] `pushEvaluationDataset(...)` follows Python's idempotent create/update/import flow: create first, update only on 409, import cases with `onCaseConflict`, skip import for empty datasets, return metadata only.
- [ ] `pushEvaluationDataset(...)` accepts explicit `inputSchema`, `outputSchema`, and `metadataSchema` options with `undefined` = omit, object = set, `null` = clear.
- [ ] `pushEvaluationDataset(...)` serializes dataset-level, report-level, and case-level evaluators to hosted `{ name, arguments }` specs.
- [ ] `pushEvaluationDataset(...)` sends empty evaluator arrays on create/update/import where needed so re-pushing can clear removed evaluators.
- [ ] `pushEvaluationDataset(...)` validates and normalizes pushed values to JSON-compatible data, including `Date` to ISO strings, with a constrained `serializeValue` escape hatch for unsupported domain values.
- [ ] `getEvaluationDataset(...)` decodes hosted evaluator specs into known/custom evaluator instances using the existing evals registry/deserialization model.
- [ ] `getEvaluationDataset(...)` has evaluator-bearing export fixture coverage for dataset-level, report-level, and case-level evaluator specs.
- [ ] `getEvaluationDataset(...)` throws on unknown evaluator specs by default; no skip or placeholder mode is added in v1.
- [ ] `getEvaluationDataset(...)` supports library-agnostic parser hooks for inputs, expected outputs, and metadata.
- [ ] `getEvaluationDataset(...)` drops hosted-only case IDs, tags, and timestamps when creating local `Case` instances.
- [ ] Docs explain the distinction between raw hosted dataset JSON and local executable evaluation `Dataset` objects.
- [ ] Docs explain schema options, parser hooks, custom evaluators, named-case idempotency, and trusted-runtime API key usage.
- [ ] A minor changeset is added for both `logfire` and `@pydantic/logfire-node`.
- [ ] Unit tests use injected/mocked `fetch`; no live Logfire service is required.

### Clarified Decisions

- API placement: high-level hosted evaluation dataset bridge lives on `LogfireAPIClient`, following Python's client-owned model.
- Public high-level names: use explicit TypeScript-readable methods `pushEvaluationDataset(...)` and `getEvaluationDataset(...)`.
- Keep low-level hosted JSON methods separate: `getDataset(...)`, `createDataset(...)`, `addCases(...)`, etc. continue to mean raw hosted dataset resource operations.
- Do not add ambiguous `pushDataset(...)` / overloaded `getDataset(..., { as: 'evals' })` aliases in v1.
- Schema handling for `pushEvaluationDataset`: accept explicit `inputSchema`, `outputSchema`, and `metadataSchema` options. `undefined` omits/leaves unchanged on update, object sets, `null` clears.
- No built-in Zod/Valibot/etc adapters in v1. Users can pass JSON Schema generated by external libraries.
- `pushEvaluationDataset(...)` return value follows Python: return hosted dataset metadata from `getDataset(name, { includeCases: false })` after create/update and case import.
- `pushEvaluationDataset(...)` idempotency follows Python: create first, update only on 409 `DatasetApiError`, import cases with `onCaseConflict` defaulting to `update`, skip import when there are no cases, propagate non-409 errors.
- `getEvaluationDataset(...)` unknown evaluator behavior: strict by default and only v1 mode. Unknown case-level, dataset-level, or report-level evaluator specs throw during deserialization.
- Evaluator spec conversion: use dedicated internal hosted bridge helpers for local `Dataset`/`Case` <-> hosted API JSON conversion.
- Do not overload `Dataset.toObject()` with a hosted format and do not make low-level hosted API methods accept local file-format evaluator specs.
- Push value serialization: use B-lite default normalization plus a constrained `serializeValue(value, context)` escape hatch.
- Fetch value parsing: `getEvaluationDataset(...)` supports library-agnostic parser hooks `parseInputs`, `parseExpectedOutput`, and `parseMetadata` for case values.
- Hosted-only fields on fetch: drop hosted case IDs, tags, and timestamps when creating local `Case` instances.
- Unnamed cases: follow Python and allow them in `pushEvaluationDataset(...)`. Document that re-push idempotency applies to named cases.
- Implementation organization: no new public subpath/export for the bridge. Keep public methods and option types on `logfire/datasets`; organize internal helper files under `src/datasets/` according to code size/complexity.
- Import direction: `logfire/datasets` may statically import `logfire/evals` internals/classes for the high-level bridge.
- Runtime/security guard: no hard browser/runtime guard in v1. Treat the hosted API client as trusted-server API by documentation.
- Docs scope: update `docs/evals.md`; no README updates or runnable examples in v1 unless implementation discovers an existing required pattern.
- Release note: add a new minor changeset for `logfire` and `@pydantic/logfire-node`.

### Python Reference

- `../logfire/logfire/experimental/api_client.py`
  - `LogfireAPIClient.push_dataset(...)`
    - target name = `dataset.name` unless `name=` override is passed;
    - raises when neither local dataset name nor override is present;
    - builds create/update kwargs with schemas, description, dataset evaluators, and report evaluators;
    - calls `create_dataset(...)`;
    - on `DatasetApiError.status_code == 409`, calls `update_dataset(...)`;
    - calls `add_cases(..., on_conflict=on_case_conflict)` when cases exist;
    - returns `get_dataset(target_name, include_cases=False)`.
  - `LogfireAPIClient.get_dataset(...)`
    - no type args: returns raw export dict;
    - `include_cases=False`: returns metadata only;
    - with input/output/metadata types: returns `pydantic_evals.Dataset`;
    - accepts `custom_evaluator_types` and `custom_report_evaluator_types`.
  - `_serialize_evaluators(...)`
    - hosted shape is `[{ name: string, arguments: object | null }]`.
  - `_serialize_case(...)`
    - serializes inputs, expected output, metadata, and case-level evaluators.
  - `_build_push_dataset_kwargs(...)`
    - always passes dataset-level `evaluators` and `report_evaluators`, including empty sequences, so re-pushing can clear hosted values.

  - Do not feed raw `{ name, arguments }` objects directly into `Dataset.fromObject(...)`; JS will treat them as malformed single-key evaluator encodings.

### Gotchas

- TypeScript generics are erased. Do not attempt Python-style schema inference from `Dataset<I, O, M>`.
- Do not accept Zod/Valibot schema objects directly. Accept JSON Schema for hosted schemas and parser functions for fetched values.
- Raw hosted JSON and local executable `Dataset` objects are different representations. Keep names explicit.
- Hosted evaluator spec shape is `{ name, arguments }`; local file/YAML evaluator shape is `string | { [Name]: args }`.
- The backend `/export/` evaluator shape is load-bearing for `getEvaluationDataset(...)` and is not well-covered by Python fixtures. Before implementation, verify the backend schema or use tests to document the chosen fixture shape. For robustness, the high-level fetch converter may accept both explicit hosted specs and compact file/YAML evaluator specs, but push must send explicit hosted specs.
- `decodeSpec({ SomeEvaluator: null })` means a single positional `null`; no-argument hosted specs with `arguments: null` must map to bare string `'SomeEvaluator'`.
- Empty evaluator arguments should be folded to `arguments: null` on push, matching Python's `_serialize_evaluators(...)` behavior for empty dicts and local `encodeEvaluatorSpec(...)` behavior for empty objects/arrays.
- Avoid multi-positional hosted evaluator arguments in v1. If an evaluator `toJSON()` returns an array with more than one element, fail with a clear error unless implementation verifies backend/Python compatibility for that shape.
- Unknown evaluator specs must fail during `getEvaluationDataset(...)`, not produce placeholder/fake evaluators.
- `serializeValue` is an escape hatch for unsupported values only. Its return value must be re-normalized/revalidated; it must not bypass JSON safety.
- `undefined` is dangerous in JSON: object properties disappear and array entries become `null`. Reject unsafe `undefined` inside provided values with a path-aware error.
- The JS normalizer is intentionally stricter than Python's Pydantic serialization for nested values. Values that Python may serialize automatically, such as dataclasses or class instances, must be converted by JS users through JSON-compatible values or `serializeValue`.
- Re-push idempotency depends on stable case names. Unnamed cases are allowed, matching Python, but may not be idempotent.
- Hosted exports with duplicate named cases should surface the existing local `Dataset` duplicate-name error. The backend is expected to enforce case-name uniqueness; no special dedupe logic belongs in this bridge.
- Do not add runtime/browser guards. Rely on docs for trusted-server usage and API key handling.
- Do not create a new public import path such as `logfire/datasets/evaluation`.

## Recommended Public API Shape

```ts
import { createLogfireAPIClient } from '@pydantic/logfire-node/datasets'
import { Case, Dataset, EqualsExpected } from 'logfire/evals'

const dataset = new Dataset<{ question: string }, { answer: string }>({
  cases: [
    new Case({
      expectedOutput: { answer: '4' },
      inputs: { question: 'What is 2+2?' },
      name: 'arithmetic-1',
    }),
  ],
  evaluators: [new EqualsExpected()],
  name: 'qa-golden-set',
})

const client = createLogfireAPIClient()

const hosted = await client.pushEvaluationDataset(dataset, {
  description: 'Golden arithmetic cases',
  inputSchema: {
    properties: { question: { type: 'string' } },
    required: ['question'],
    type: 'object',
  },
  outputSchema: {
    properties: { answer: { type: 'string' } },
    required: ['answer'],
    type: 'object',
  },
})

console.log(hosted.id, hosted.case_count)

const fetched = await client.getEvaluationDataset<{ question: string }, { answer: string }>('qa-golden-set')
await fetched.evaluate(async ({ question }) => ({ answer: question.includes('2+2') ? '4' : 'unknown' }))
```

Parser hook example:

```ts
const fetched = await client.getEvaluationDataset<MyInput, MyOutput, MyMetadata>('qa-golden-set', {
  parseExpectedOutput: (value) => OutputSchema.parse(value),
  parseInputs: (value) => InputSchema.parse(value),
  parseMetadata: (value) => MetadataSchema.parse(value),
})
```

Custom value serialization example:

```ts
await client.pushEvaluationDataset(datasetWithSets, {
  serializeValue(value) {
    if (value instanceof Set) {
      return Array.from(value)
    }
    return undefined
  },
})
```

Raw hosted JSON remains separate:

```ts
const rawExport = await client.getDataset('qa-golden-set')
const metadataOnly = await client.getDataset('qa-golden-set', { includeCases: false })
const cases = await client.listCases('qa-golden-set')
```

## Implementation Blueprint

### Data Models

Add public option/context types to `logfire/datasets`.

Recommended shape:

```ts
export interface EvaluationDatasetValueContext {
  caseIndex?: number
  caseName?: string
  field: 'evaluatorArguments' | 'expectedOutput' | 'inputs' | 'metadata'
  path: string
}

export type EvaluationDatasetValueSerializer = (value: unknown, context: EvaluationDatasetValueContext) => unknown

export type EvaluationDatasetValueParser<T> = (value: unknown, context: EvaluationDatasetValueContext) => T

export interface PushEvaluationDatasetOptions {
  description?: null | string
  inputSchema?: JsonSchema | null
  metadataSchema?: JsonSchema | null
  name?: string
  onCaseConflict?: CaseConflictBehavior
  outputSchema?: JsonSchema | null
  serializeValue?: EvaluationDatasetValueSerializer
}

export interface GetEvaluationDatasetOptions<Inputs = unknown, Output = unknown, Metadata = unknown> {
  customEvaluators?: readonly EvaluatorClass<Inputs, Output, Metadata>[]
  customReportEvaluators?: readonly ReportEvaluatorClass<Inputs, Output, Metadata>[]
  parseExpectedOutput?: EvaluationDatasetValueParser<Output>
  parseInputs?: EvaluationDatasetValueParser<Inputs>
  parseMetadata?: EvaluationDatasetValueParser<Metadata>
  primaryArgKeys?: Record<string, string>
}
```

Notes:

- Import `Dataset`, `Case`, `EvaluatorClass`, and `ReportEvaluatorClass` from `../evals`.
- `primaryArgKeys` is intentionally public because it mirrors existing `Dataset.fromObject(...)` / `FromOptions` behavior and is needed for custom single-positional evaluator specs. Keep it documented as an advanced compatibility option, not the primary path.
- Parser hooks should apply only to case values, not evaluator specs.
- Do not expose internal hosted conversion helpers as package exports.
